Background Loops
Fast Sweep (1 s)
Discovers workflows with next_poll_at ≤ now or status = pending. Drives the claim-poll-release cycle for async steps.
Heartbeat (TTL/3)
Extends lock TTL on all workflows claimed by this instance. If the process dies, the lock expires after TTL seconds.
Slow Sweep (30 s)
Detects anomalies: steps stuck in SUBMITTED or RUNNING without a lock; completed steps with un-advanced index; stale locks past TTL.
Fence Token (Optimistic Lock)
fence_token: N
Incremented atomically on every try_claim. All subsequent writes carry {"fence_token": N} as a filter — stale writers silently fail.
findOneAndUpdate
{status: "pending"|"running", locked_by: null, fence_token: N-1} → {$set: {locked_by: id}, $inc: {fence_token: 1}}

State Transitions
Step Status
FromToTrigger
PENDING → SUBMITTED write-ahead before exec
SUBMITTED → RUNNING handler starts
RUNNING → COMPLETED sync handler returns
RUNNING → BLOCKED async handler returns
BLOCKED → COMPLETED completeness_check true
RUNNING → FAILED retries exhausted
BLOCKED → FAILED poll timeout / max_polls
Workflow Status
FromToTrigger
PENDING → RUNNING try_claim succeeds
RUNNING → COMPLETED last step completes
RUNNING → FAILED step fails after retries
RUNNING → NEEDS_REVIEW crash, no recovery path

All Retries Exhausted lock released

Crash Recovery Path
Crash During Step Execution
Process dies mid-step. Lock TTL expires. Slow sweep detects orphaned step in SUBMITTED or RUNNING state without a live lock.
Recovery Decision Tree
1. Has verify_completion hook?
YES → Call it. If returns True → mark COMPLETED, advance.
NO ↓
2. Is async step with completeness_check?
YES → Run completeness_check. If complete → COMPLETED. Else → set BLOCKED, resume polling.
NO ↓
3. Is step handler idempotent?
YES → Re-execute handler from scratch. Safe because idempotent.
NO ↓
4. Fallback
NEEDS_REVIEW → workflow.status → "needs_review". Requires human intervention.
