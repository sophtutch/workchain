Workflow Engine — Flow Diagram

Sweep-only discovery • Claim-poll-release for async steps • MongoDB as sole state store • Multi-instance safe

Example Workflow
User Onboarding
1. validate_input 2. create_account 3. provision_resources 4. send_welcome_email
Background Loops
Fast Sweep
Polls MongoDB for new, abandoned, and released async workflows. Claims atomically with fence token.
every 5s → find_claimable()
Heartbeat
Extends lock TTL on active workflows. Cancels task if lock is stolen.
every 10s → heartbeat()
Slow Sweep
Detects stuck steps, stale locks, un-advanced indexes. Force-releases locks.
every 60s → find_anomalies()
Fence Token
Incremented on every try_claim(). Every write includes the token in its query filter. If the token doesn't match, the write is silently rejected — the instance lost its lock.
Instance A claims → fence=3
Lock expires, B claims → fence=4
A tries to write (fence=3) → rejected
Discovery
Engine — Fast Sweep
find_claimable() returns workflow ID.
try_claim() → atomic findOneAndUpdate sets locked_by, extends TTL, $inc fence_token.
Status: pending → running

Retry Observability
Each attempt is persisted before execution via _run_step_with_retry(). Query MongoDB at any time to see:
steps.1.attempt = 3, steps.1.status = "running"
If the process crashes mid-attempt, the DB shows exactly which retry it was on.

If All Retries Exhausted
Retry Failure Path
When all max_attempts are exhausted, tenacity re-raises the final exception:
attempt 5/5 → exception raised
engine catches → persist step.status = FAILED
             → persist step.result.error = traceback
             → workflow.status = FAILED
             → release lock, remove from _active
If Process Dies Mid-Step
Crash Recovery Path
Step found in SUBMITTED or RUNNING when a new instance claims the workflow:
verify_completion hook? → yes → call it → done? skip
                      ↓ no
async + completeness_check + has result?
  → yes → call checker → complete? mark COMPLETED
  → yes → call checker → not complete? → BLOCKED (don't re-submit)
                      ↓ no / threw
step is idempotent?    → yes → reset to PENDING, re-run
                      ↓ no
mark workflow NEEDS_REVIEW, stop
